Çok yerinde bir soru — ve net cevabı şu: Atomik UPDATE over-engineering değildi, hatta saf performans/güvenlik açısından üçü içinde en iyisiydi. Ama bizim çözümün mimarisine ve case'in ölçtüğü şeye pessimistic'ten daha kötü uyuyordu.

Önce küçük bir düzeltme: biz aslında pessimistic ile ilerledik; optimistic (@Version) sadece ucuz bir ikinci savunma katmanı olarak dormant duruyor, birincil mekanizma o değil. Yani gerçek seçim "pessimistic mi, atomik mi" idi.

Atomik UPDATE nedir, neden bu kadar iyi?
UPDATE products SET stock = stock - :qty WHERE id = :id AND stock >= :qty
Tek satır. Veritabanı bunu tek atomik işlem olarak yapar: ya stock >= qty'dir ve düşer (1 satır etkilenir), ya değildir ve hiç dokunmaz (0 satır). Oversell fiziksel olarak imkânsız, lock tutmaz, tam concurrent, en hızlısı. Korkutucu görünür ama aslında en sade ve en bulletproof olanı.

O zaman neden seçmedik? 4 gerçek neden (hiçbiri "daha kötü çalışır" değil)
1. JPA persistence context'ini bypass eder — entity yalan söyler.
@Modifying @Query ile çalışan UPDATE doğrudan DB'ye gider. Elindeki managed Product nesnesinin stockQuantity alanı hâlâ eski değeri gösterir, @Version JPA üzerinden artmaz. Düzeltmek için clearAutomatically=true/flushAutomatically=true + gerekirse yeniden okuma gerekir. "İki ayrı doğruluk kaynağı" (DB gerçeği vs. bellekteki bayat entity) sinsi bug üretir.

2. Az önce kurduğumuz rich domain model'i geri bozar.
Daha yeni invariant'ı entity'ye taşıdık: Product.decreaseStock() kuralı kendi içinde tutuyor ve InsufficientStockException fırlatıyor. Atomik UPDATE bu kuralı tekrar SQL'e/repository'ye iter; entity kendi kuralını koruyamaz, "etkilenen satır == 0 → throw" kontrolünü service'e yazarsın. Bu, tam da kaçındığımız anemic model yönüdür. Yani çözümün geri kalanının SOLID/DDD duruşuyla çelişir.

3. Case'in ölçtüğü şeyi daha az gösterir.
Case açıkça "transaction yönetimini ve race condition'ı anlıyor musun" diye bakıyor. Atomik UPDATE tüm sorunu DB motoruna devreder — doğru ama "DB hallediyor" cevabıdır. Pessimistic lock ise locking, isolation, lock ordering ile deadlock önleme üzerine düşündüğünü gösterir; mülakatta "burada neden bu yolu seçtin" sorusuna anlatacak çok daha zengin malzeme verir. Junior pozisyonunda yetkinlik kanıtlamak için bu daha değerli.

4. Aslında işi azaltmaz, hibrit'e zorlar.
Atomik UPDATE sadece stoğu çözer. Sipariş + ödeme + rollback bütünlüğü için yine de etrafına @Transactional koymak zorundasın. Adım 1 — Mimari ve bir sipariş isteğinin uçtan uca akışı
Katmanlar ve görevleri
Katman (paket)	Sorumluluk	"Neden ayrı?"
api/controller	HTTP sınırı. Girdiyi doğrula, DTO→command çevir, servisi çağır, sonucu DTO'ya çevir. İş kuralı yok.	Controller değişikliği (HTTP biçimi) iş mantığını etkilemesin
api/dto (request/response)	Tel sözleşmesi (JSON). Validation anotasyonları burada.	Entity'ler dışarı sızmasın → DB şeması API'ye bağlı kalmasın
api/mapper	Entity/command ↔ DTO çevirisi (elle)	Çeviri tek yerde, test edilebilir
api/exception	GlobalExceptionHandler + ApiError	Tutarlı, tek-tip hata sözleşmesi
service	Use-case orkestrasyonu + transaction sınırı. CreateOrderCommand servis girdisi	İş akışının sahibi; web ve DB'den bağımsız
domain/model	JPA entity'leri + invariant'lar (Product.decreaseStock, OrderItem.of, Order.recalculateTotal/markConfirmed)	Kurallar entity'de yaşar (anemic değil) → bypass edilemez
payment	Strategy alt sistemi (interface + registry + 3 strateji)	Ödeme bağımsız bir modül; OCP ile genişler
repository	Spring Data JPA. ProductRepository.findByIdForUpdate (pessimistic lock), OrderRepository (entity graph)	Veri erişimi soyutlanır
config	AsyncConfig (executor), WebConfig/CorsProperties, PaymentProperties, DataInitializer	Kros-kesit yapılandırma tek yerde
exception (kök)	İş istisnaları (InsufficientStock, PaymentFailed, *NotFound)	Domain, api'ye bağımlı olmadan kullanabilsin
Bağımlılık yönü (tek yön): Controller → Service → Repository; Service → payment (registry üzerinden); domain hiçbir şeye bağımlı değil. Entity'ler hiçbir zaman controller'a çıkmaz (hep DTO).

Uçtan uca: POST /api/orders bir istek
HTTP POST /api/orders {customerId, paymentMethod, items[]}
   │
1. OrderController.createOrder(@Valid CreateOrderRequest)
   │   └─ Bean Validation: customerId/paymentMethod NotNull, items NotEmpty,
   │      quantity 1..100000 → ihlal: MethodArgumentNotValidException → 400
2. OrderMapper.toCommand(req) → CreateOrderCommand   (web DTO'sundan kopar)
   │
3. OrderService.createOrder(cmd)   ┌── @Transactional başlar ──────────────┐
   │   a. Order(PENDING, total=0) kur                                       │
   │   b. kalemleri productId'ye göre ARTAN sırala  (deadlock önleme)       │
   │   c. her kalem için:                                                   │
   │        productService.getForUpdate(id)                                 │
   │          → ProductRepository.findByIdForUpdate  (SELECT … FOR UPDATE)  │
   │            = pessimistic write lock → aynı ürün çekişmesini serialize  │
   │        product.decreaseStock(qty)  → yetersizse InsufficientStock 💥   │
   │        OrderItem.of(product, qty)  → fiyat snapshot + subtotal         │
   │        order.addItem(item)                                             │
   │   d. order.recalculateTotal()                                          │
   │   e. paymentService.charge(method, total)                             │
   │        → registry.resolve(method) → strategy.pay()                     │
   │          başarısız (ör. CC limit) → PaymentFailed 💥                   │
   │          başarılı → COMPLETED Payment; order.attachPayment(p)          │
   │   f. order.markConfirmed()                                             │
   │   g. orderRepository.save(order)  (items + payment cascade)            │
   │   h. notificationService.notifyOrderConfirmed(order)  (log)            │
   │                              └── herhangi bir 💥 → TÜM transaction rollback ┘
4. OrderMapper.toResponse(order) → OrderResponse → HTTP 201
Kritik nokta (mülakatta vurgula): a–h'nin tamamı tek transaction. c'de stok düşüşü veya e'de ödeme patlarsa, o ana kadar bellekte yapılmış stok düşüşleri commit edilmez → ne yarım sipariş, ne değişmiş stok. Hata GlobalExceptionHandler'da doğru HTTP koduna çevrilir (409/422).

Neden bu katmanlama? Her katmanın tek bir değişme sebebi var (SRP): HTTP sözleşmesi değişirse api, iş akışı değişirse service, kural değişirse domain, ödeme yöntemi eklenirse sadece payment. Bu izolasyon olmasaydı "her şey OrderService'te" olur, test edilemez ve kırılgan olurdu.

🎤 60 saniyelik asansör konuşması (mülakat açılışı)
"Bu, küçük bir e-ticaret sisteminin sipariş işleme servisi — Java 17 / Spring Boot 3, in-memory H2 ile tek komutla çalışıyor. Müşteri sipariş verince servis stoğu kontrol eder, siparişi oluşturup stoğu düşer, seçilen yönteme göre ödemeyi alır ve müşteriyi bilgilendirir — hepsi atomik olarak. Üç mühendislik odağı var: (1) Eşzamanlılık — aynı ürüne paralel siparişler asla fazla satış yapamaz; bunu pessimistic row lock ve tutarlı kilit sırasıyla çözdüm. (2) Transaction — sipariş + stok + ödeme tek transaction; herhangi biri patlarsa her şey geri sarılır. (3) Design pattern — ödeme yöntemleri Strategy + Registry ile; yeni yöntem eklemek sadece yeni bir component (Open/Closed). Katmanlı bir yapı (controller → service → domain/repository), kritik eşzamanlılık ve rollback yollarını kanıtlayan 39 test, Maven veya Docker ile çalışma ve API'yi denemek için statik bir test paneli içeriyor."

Diyelim istek: { customerId: 42, paymentMethod: "CREDIT_CARD", items: [ {productId:1, qty:2}, {productId:2, qty:1} ] }

① OrderController.createOrder(@Valid CreateOrderRequest)
     • Bean Validation ÖNCE koşar (customerId/paymentMethod NotNull, items NotEmpty,
       qty 1..100000). İhlal → MethodArgumentNotValidException → handler → 400 (DB'ye hiç gidilmez)
     • @ResponseStatus(CREATED) → başarıda 201
② OrderMapper.toCommand(req) → CreateOrderCommand
     • Web DTO'sunu servis dilinden ayırır (servis HTTP'yi bilmez)
③ OrderService.createOrder(cmd)   ╔═══ @Transactional: proxy girişte transaction AÇAR ═══╗
     a) Order kur: status=PENDING, totalAmount=0                                          ║
     b) kalemleri productId'ye göre ARTAN sırala → [ürün1, ürün2]                         ║
        (deadlock önleme: herkes aynı sırada kilitlerse döngü oluşmaz)                    ║
     c) HER kalem için:                                                                   ║
        ┌ productService.getForUpdate(id)                                                 ║
        │   └ ProductRepository.findByIdForUpdate(id)                                     ║
        │       → SQL: SELECT … FROM products WHERE id=? FOR UPDATE                       ║
        │       → satıra PESSIMISTIC WRITE LOCK; aynı ürüne paralel başka                 ║
        │         transaction varsa burada BEKLER (serialize olur)                        ║
        ├ product.decreaseStock(qty)   ← invariant entity'de                              ║
        │   → stok < qty ise InsufficientStockException 💥 (rollback'e gider)             ║
        │   → değilse stoğu bellekte düşer (managed entity; commit'te UPDATE flush olur)  ║
        └ order.addItem( OrderItem.of(product, qty) )                                     ║
            → OrderItem.of: unitPrice = o anki fiyat (snapshot), subtotal = fiyat×qty     ║
     d) order.recalculateTotal()  → Σ subtotal = 2×1499.99 + 1×29.90 = 3029.88            ║
     e) paymentService.charge(CREDIT_CARD, 3029.88)                                       ║
        └ registry.resolve(CREDIT_CARD) → CreditCardPaymentStrategy                       ║
            → limit kontrolü: 3029.88 ≤ 10000 → onay                                      ║
            → PaymentResult.success(ref)                                                  ║
          başarısız olsa → PaymentFailedException 💥 (rollback)                           ║
          başarılı → Payment(COMPLETED, ref); order.attachPayment(payment)                ║
     f) order.markConfirmed()  → status=CONFIRMED                                         ║
     g) orderRepository.save(order)                                                       ║
        → cascade ile OrderItem'lar + Payment de yazılır                                  ║
     h) notificationService.notifyOrderConfirmed(order)  → log                            ║
                          ╚═══ metot normal dönerse: COMMIT (stok UPDATE'i + insert'ler) ═╝
                              herhangi bir 💥 unchecked exception → ROLLBACK (hiçbiri kalıcı olmaz)
④ OrderMapper.toResponse(order) → OrderResponse → HTTP 201
   (hata olduysa GlobalExceptionHandler: InsufficientStock→409, PaymentFailed→422)
Üç can alıcı noktayı netleştirelim
Stok kilitleme (c): Kilit, findByIdForUpdate'in SELECT … FOR UPDATE'iyle alınır ve transaction commit/rollback olana kadar tutulur. Yani aynı ürüne 100 paralel istek gelse, hepsi bu satırda sıraya girer; biri işini bitirip commit edene kadar diğeri bekler. Stok bitince bekleyenler decreaseStock'ta InsufficientStockException alır — temiz "yetersiz stok", lock hatası değil.

Ödeme adımı (e): Ödeme, transaction'ın içinde. Burada gerçek bir dış gateway yok (case öyle istemiyor), mock bir strateji. Gerçek gateway olsaydı bunu transaction'ın içinde tutmazdık (uzun süren dış çağrı transaction'ı kilitli tutar) — orada Saga/outbox düşünürdük; bunu "ölçeklenince" kısmında konuşacağız.

Transaction sınırı: Sınır tam olarak OrderService.createOrder proxy'sinde. Transaction girişte açılır, normal dönüşte commit edilir. Tüm iş istisnalarımız RuntimeException türevi olduğu için Spring default rollback yapar — rollbackFor yazmayı unutma riski yok. c'de bellekte düşülen stok bile commit edilmediği için DB'de hiç değişmez.

Bir incelik: createOrder controller'dan (ayrı bean) çağrıldığı için Spring proxy devrede ve @Transactional çalışır. Aynı sınıf içinden çağrılsaydı (self-invocation) proxy atlanır ve transaction hiç başlamazdı — BulkOrderService'i ayrı bean yapmamızın sebebi de bu (Adım 2'de açacağız).

İskelet bu. İstersen burada birkaç soruyla pekiştirelim (ör. "neden ayrı CreateOrderCommand?", "entity neden response'a çıkmıyor?", "@EntityGraph neden var?"), yoksa Adım 2 — üç odak alanı (trade-off Q&A'larıyla: pessimistic vs optimistic vs atomic, paralel izolasyon, proxy tuzağı) ile devam edelim. Nasıl istersin?

Mimari ve akış çok net oturdu. Şimdi Adım 2. Üç odak alanını derinlemesine konuşalım. Sırayla: paralel işlem (BulkOrderService neden ayrı bean, bulk neden @Transactional değil, CompletableFuture + bounded thread pool, hata izolasyonu nasıl sağlanıyor), sonra transaction yönetimi (tek @Transactional sınırı, unchecked exception ile default rollback, self-invocation/proxy tuzağı), sonra design pattern (Strategy + Registry, neden bu pattern, Open/Closed). Her başlıkta "neden böyle, alternatifi neydi" trade-off'larını da göm. özellikle pessimistic vs optimistic vs atomic update karşılaştırması, ve bulk'ta neden 200 + per-item sonuç döndüğümüz. Mülakatta en çok bu üç başlıktan soru geleceği için sağlam ve örnekli anlat. Kendi katmak istediğin şeyler varsa onları da katarak adım 2 ye Üç odak alanına geçelim.

İskelet net — şimdi en çok soru gelecek yere giriyoruz. Her başlıkta "nasıl çalışıyor → neden böyle, alternatifi neydi → kanıt/örnek" sırasını izleyeceğim.

Adım 2 — Üç Odak Alanı
A) Paralel İşlem
Önce problemi görelim: neden kilit gerekiyor?
Stok=10 bir ürüne iki thread aynı anda 3'er adet sipariş etsin. Kilit yoksa:

T1 okur stok=10   T2 okur stok=10
T1 yazar 10-3=7   T2 yazar 10-3=7     → final 7, oysa 4 olmalı (lost update)
Stok 0'a yakınken bu oversell demek. Case'in test edeceği tam senaryo bu. Çözüm: aynı ürüne erişimi serialize etmek.

BulkOrderService neden ayrı bean?
processBulkOrders her siparişi ayrı thread'de orderService.createOrder(...) çağırıyor. createOrder @Transactional. Transaction Spring'de proxy ile çalışır — bean'e dışarıdan girişte devreye girer. Eğer paralel dağıtımı OrderService'in kendi içine koysaydım, this.createOrder(...) çağrısı kendi proxy'sini atlardı (self-invocation) ve transaction hiç başlamazdı. Ayrı bean olunca BulkOrderService, enjekte edilmiş OrderService proxy'sini çağırır → her görev gerçek bir transaction açar. (Bu, "proxy tuzağı"nın somut örneği; B bölümünde tekrar değineceğiz.)

İkinci sebep SRP: tek-sipariş use-case'i (OrderService) ile paralel-orkestrasyon (BulkOrderService) farklı sorumluluklar.

Bulk neden @Transactional DEĞİL?
Transaction thread-bound'dur (Spring transaction context'i bir ThreadLocal'da tutulur, thread'ler arası yayılmaz). Bulk metoduna @Transactional koysaydım:

O transaction sadece bulk'ı çağıran thread'de açılırdı; CompletableFuture görevleri başka thread'lerde koştuğu için o transaction'a katılmazlardı → faydasız, üstelik yanıltıcı.
Asıl istediğimiz izolasyon: her sipariş kendi transaction'ında açılıp commit/rollback olsun ki birinin rollback'i diğerini etkilemesin. Bunu, her görevin executor thread'inde kendi createOrder transaction'ını açmasıyla elde ediyoruz. Yani bulk'ı kasıtlı olarak transaction'sız bıraktım; transaction sınırı tek sipariş seviyesinde.
CompletableFuture + bounded thread pool
CompletableFuture.supplyAsync(() -> processOne(index, command), executor)
commonPool() değil, özel executor (AsyncConfig): core 10 / max 20 / queue 50, CallerRunsPolicy. Neden? supplyAsync'in default'u JVM-wide ForkJoinPool.commonPool()'dur; bizim görevlerimiz bloklayan JDBC işi yapıyor (pessimistic lock'ta bekleme dahil). Ortak havuzu doldurursak JVM'deki başka işleri (paralel stream'ler vb.) aç bırakırız. Bounded havuz hem izolasyon hem back-pressure verir.
CallerRunsPolicy: queue+pool dolarsa görevi reddetmek yerine çağıran thread çalıştırır → ani yükte sipariş düşmez, doğal olarak yavaşlar. Alternatif AbortPolicy olurdu (reddeder, sipariş kaybı) — istemeyiz.
Hata izolasyonu nasıl?
private BulkOrderResult processOne(int index, CreateOrderCommand cmd) {
    try { return BulkOrderResult.success(index, orderService.createOrder(cmd).getId()); }
    catch (Exception e) { return BulkOrderResult.failed(index, e.getMessage()); }
}
Her görev hatayı kendi içinde yakalar ve bir sonuç objesine çevirir. Böylece:

join() patlamaz (exception future'dan dışarı sızmaz),
bir siparişin rollback'i kendi transaction'ıyla sınırlı kalır,
yanıt her sipariş için {index, status, orderId|error} taşır.
🔑 Trade-off: Pessimistic vs Optimistic vs Atomic Update
Mülakatta en kritik karşılaştırma. Test senaryosu tek hot row'a yüzlerce eşzamanlı istek:

Pessimistic (seçtiğim)	Optimistic (@Version + retry)	Atomic UPDATE
Mekanizma	SELECT … FOR UPDATE, satır kilidi	commit'te version çakışması → retry	UPDATE … WHERE stock>=:q
Hot row davranışı	Temiz serialize	Retry storm	Tam concurrent, en hızlı
Hata tipi	İş hatası (yetersiz stok)	OptimisticLockException (lock hatası)	İş hatası (0 satır)
Test determinizmi	✅ kesin	❌ flaky, retry tükenmesi	✅ kesin
JPA uyumu	✅ tam	✅ tam	❌ context'i bypass eder, entity bayatlar
Domain modeli	✅ invariant entity'de	✅	❌ kural SQL'e kaçar
Optimistic'i neden reddettim: Tek satırda 100 istek → her turda 1 kazanır, 99'u version çakışıp OptimisticLockException alır. Retry gerekir; maxAttempts=3 ile çoğu tükenip false-fail olur — yani stok varken bile "hata" döner, bug gibi görünür. Düşük çekişmede optimistic mükemmel; ama burada tam test edilen noktada en kırılgan. (Bonus: optimistic'i retry'la kursaydık @Retryable'ı @Transactional'ın dışına sarmak gerekirdi — aynı metoda koyarsan kirlenmiş transaction içinde retry'lar, işe yaramaz. Pessimistic seçince bu tuzak da yok oldu, retry bağımlılığını bile kaldırdık.)
Atomic UPDATE neden değil: Aslında en sağlam/performanslı o. Ama JPA persistence context'ini bypass eder (bellekteki Product bayatlar, @Version JPA üzerinden artmaz), invariant'ı domain'den SQL'e taşır. Bizim hedef rich domain + transaction/locking anlayışını göstermek; ayrıca çok-ürünlü siparişte yine de etrafına transaction gerekir → hibrit olur. Ne zaman atomic? Çok sayıda farklı ürünlü, yüksek-throughput gerçek production envanteri (bunu "ölçeklenince" kısmında konuşacağız).
Deadlock-freedom (kilit sırası)
Çok-kalemli siparişte iki transaction ters sırada kilitlerse: T1: A→B, T2: B→A → T1 A'yı tutup B'yi bekler, T2 B'yi tutup A'yı bekler = deadlock. Çözüm: createOrder kalemleri artan productId sırasına sokuyor, herkes aynı sırada kilitliyor → döngü oluşamaz. OrderedLockingConcurrencyTest bunu kanıtlıyor (ters-sıralı 100 paralel sipariş, deadlock yok).

🔑 Trade-off: Bulk neden 200 + per-item sonuç, 207 değil?
Toplu işlemin kendisi başarılı (çalıştı, hepsini işledi); tekil başarısızlıklar bir hata değil, veridir. Bunu yapısal gövdede taşıyorum: {total, succeeded, failed, results[]}. 207 Multi-Status semantik olarak savunulabilir ama: çoğu HTTP client/proxy 207'yi alışılmadık karşılar, hata gibi ele alabilir; faydası marjinal. 200 + net per-item daha öngörülebilir ve client'ı şaşırtmaz. (Bunu README'de "değerlendirilen alternatif" olarak da not düştüm — bilinçli karar olduğunu gösterir.)

B) Transaction Yönetimi
Tek @Transactional sınırı
Sınır tam olarak OrderService.createOrder. Stok düşürme + sipariş/kalem kaydı + ödeme + bildirim hepsi bu sınırın içinde. Transaction girişte açılır, normal dönüşte commit edilir. c adımındaki pessimistic kilit de commit/rollback'e kadar tutulduğu için, "kilitle → işi yap → bitir" tek tutarlı modelde ilerler.

Unchecked exception + default rollback
Spring @Transactional default olarak yalnızca RuntimeException (unchecked) türevlerinde rollback yapar; checked exception'da yapmaz. Bütün iş istisnalarım (InsufficientStockException, PaymentFailedException, *NotFoundException) RuntimeException türevi → otomatik rollback.

Alternatif/tuzak: Checked exception kullansaydım @Transactional(rollbackFor = Exception.class) yazmayı unutursam, hata fırlatılır ama transaction commit edilir → yarım sipariş/değişmiş stok = sessiz felaket. Unchecked tercihi bu tuzağı baştan kapatıyor.

OrderServiceRollbackTest (ödeme mock'la patlatılır) ve MultiItemOrderTest (ikinci kalem stoksuz) ikisi de "hata → hiçbir şey kalıcı değil"i kanıtlıyor — ikincisi ilk kalemin bellekte düşülen stoğunun bile geri sarıldığını gösteriyor.

Self-invocation / proxy tuzağı
@Transactional (ve @Async, @Cacheable …) proxy ile çalışır: Spring bean'i sarmalar, çağrı proxy'den geçince davranış eklenir. Ama aynı sınıf içinden this.method() çağrısı proxy'yi atlar → anotasyon hiç çalışmaz. En sık "neden transaction başlamıyor / neden retry olmuyor" hatası budur. Bizde:

createOrder controller'dan (farklı bean) çağrılıyor → proxy devrede ✅
Bulk dağıtımı OrderService içinde değil, ayrı BulkOrderService'te → orderService.createOrder enjekte edilmiş proxy üzerinden → her görev gerçek transaction ✅
Propagation: neden REQUIRED (default), REQUIRES_NEW değil?
Default REQUIRED: var olan transaction'a katılır, yoksa açar. Ödeme adımını ayrı bir REQUIRES_NEW transaction'a koymak cazip görünür ama tehlikeli: ödeme bağımsız commit edilir, sonra sipariş rollback olursa → para alındı, sipariş yok. Bu yüzden ödeme aynı transaction'da. (Gerçek dış gateway olsaydı hikâye değişirdi → Saga; "ölçeklenince".)

C) Design Pattern — Strategy + Registry
Nasıl çalışıyor?
PaymentStrategy arayüzü: getPaymentMethod() + pay(amount).
Her yöntem bir @Component implementasyonu (CreditCard/BankTransfer/Crypto).
PaymentStrategyRegistry constructor'ına Spring tüm PaymentStrategy bean'lerini List olarak enjekte eder; registry bunları EnumMap<PaymentMethod, PaymentStrategy>'e indeksler. resolve(method) çalışma zamanında doğru stratejiyi verir.
PaymentService.charge → registry.resolve(method).pay(amount) → sonucu Payment entity'sine çevirir.
Neden bu pattern? (Open/Closed)
Yeni ödeme yöntemi eklemek = yalnızca yeni bir @Component yazmak (getPaymentMethod() yeni enum değerini döndürür). Registry, service, controller, hiçbir switch değişmez. Kapalı-değişime/açık-genişlemeye (OCP) birebir uyum. (Bunu Adım 3'te canlı örnekle göstereceğiz.)

🔑 Trade-off: alternatifler
if/switch (method) PaymentService içinde: En basit görünür ama OCP ihlali — her yeni yöntemde o metodu düzenlersin, dosya şişer, her dalı ayrı test etmek zor. Case'in açıkça ödüllendirdiği "yeni yöntem = mevcut kodu değiştirme" şartını çiğner.
Factory: Stratejiyi üreten ayrı bir factory yazılabilirdi; ama Spring DI + registry zaten bunu deklaratif yapıyor (bean'leri otomatik toplayıp Map'liyor). Ayrı factory gereksiz katman olurdu.
Template Method: Ortak bir algoritma iskeleti + değişen adımlar olsaydı uygundu (validate→charge→record gibi sabit sıra). Ama bizim yöntemlerimiz tamamen bağımsız (ortak iskelet yok), o yüzden Strategy daha temiz.
Sağlamlaştırmalar
Registry constructor'ı aynı yönteme iki strateji gelirse IllegalStateException fırlatır (fail-fast, sessiz override yerine).
PaymentRegistryCompletenessTest: @EnumSource(PaymentMethod.class) ile her enum değerinin bir stratejiye çözüldüğünü garantiler → biri enum ekleyip stratejiyi unutursa test patlar (OCP güvencesi).
Over-engineering değil mi? Hayır: Strategy+Registry'de durduk. AbstractFactory, ayrı config, pluggable SPI gibi katmanlar eklemedik. Case "en az bir pattern, yerinde" diyor; bu tam o ölçüde.
Benim eklemek istediğim iki şey
PaymentResult value object'i neden var? Strateji ya başarı (referansla) ya da gerekçeli başarısızlık döndürür — boolean yerine. Böylece "normal red" (CC limiti) PaymentFailedException'a, "beklenmedik hata"dan ayrı, kontrollü biçimde çevrilir. Strateji exception fırlatmıyor; karar PaymentService'te → tek sorumluluk.
@EntityGraph (detay okuma): getOrderDetail items+payment'ı tek sorguda çeker. Transaction kapandıktan sonra DTO'ya çeviride lazy ilişkiye dokunulursa LazyInitializationException olurdu; entity graph bunu önler (N+1'i de azaltır).
İstersen bu üç başlıktan birini "mülakatçı modunda" sana sorularla deşeyim (ör. "100 yerine 10.000 paralel gelse pessimistic ne yapar?", "REQUIRES_NEW'u gerçekten istediğin bir durum var mı?"), yoksa Adım 3 — 'yeni bir ödeme yöntemi eklesek nasıl olurdu' canlı örneği'ne geçelim. Hangisi?

Üç odak alanı tamamdır. Adım 3'e geçelim: yeni bir ödeme yöntemi, mesela Apple Pay eklesek baştan sona nasıl olurdu, somut kod örneğiyle göster. Hangi tek dosyayı eklerim, getPaymentMethod ve pay nasıl olur, ve mevcut hiçbir şeyin değişmeden kaldığını yani Open/Closed'ı net göster. Registry'nin yeni stratejiyi otomatik toplayıp resolve ettiğini de anlat. Projeye kalıcı eklememize gerek yok, örnek üzerinden anlatman yeterli; ama istersen geçici ekleyip PaymentRegistryCompletenessTest'in hâlâ yeşil kaldığını da gösterebilirsin. İstersen kalıcı da olabilir. Ayrıca burada Strategy ve Registry patternlerin de nasıl çalıştığını örnek üzerinden göstermeni istiyorum.

Adım 3'e geçelim. Bunu hem anlatacağım hem de gerçekten geçici ekleyip Open/Closed'ı testle kanıtlayacağım, sonra geri alacağım (az önce yalınlaştırdığımız projeye kullanılmayan bir APPLE_PAY bırakmak tutarsız olur — gösterim için ekleyip temizleyeceğim).

Adım 3 — Yeni ödeme yöntemi (Apple Pay): baştan sona
Eklediğim tek dosya
@Slf4j
@Component                                   // ← Spring bunu otomatik bean yapar
public class ApplePayPaymentStrategy implements PaymentStrategy {

    @Override
    public PaymentMethod getPaymentMethod() {  // ← "ben APPLE_PAY'i işlerim" der
        return PaymentMethod.APPLE_PAY;
    }

    @Override
    public PaymentResult pay(BigDecimal amount) {  // ← bu yöntemin ödeme mantığı
        String ref = "AP-" + UUID.randomUUID();
        log.info("Processing APPLE_PAY payment of {} -> ref {}", amount, ref);
        return PaymentResult.success(ref);
    }
}
Artı tek satır: PaymentMethod enum'una APPLE_PAY değeri. Hepsi bu.

Değişmeyen her şey (Open/Closed kanıtı)
APPLE_PAY siparişi uçtan uca çalıştı; ama şu dosyaların hiçbirine dokunmadım:

OrderController · OrderService · PaymentService · PaymentStrategyRegistry · OrderMapper/DTO'lar · diğer 3 strateji · hiçbir test.

Yani dispatch eden (yönlendiren) hiçbir kod değişmedi; sadece yeni davranışı ekledim. Bu OCP'nin tam tanımı: genişlemeye açık, değişime kapalı.

Dürüst nüans (mülakatta bunu söylemek artı): "Tek dosya" tam doğru değil — PaymentMethod enum'una da bir değer ekledim. Ama bu additive bir değişiklik (tip-güvenli sözleşmeye yeni bir kelime); davranış/mantık içeren hiçbir yeri düzenlemedim. Kritik fark: ortada güncellenmesi gereken bir switch (method) yok. switch tabanlı tasarımda her yeni yöntemde o mantığı düzenlerdin (değişime kapalı değil); burada sadece yeni bir vokabüler + ayrı dosyada davranışı bildiriyorsun.

Strategy nasıl çalıştı?
PaymentStrategy bir sözleşme: "hangi yöntemim" (getPaymentMethod) + "nasıl öderim" (pay). Her yöntem bu sözleşmeyi kendi dosyasında uygular. Çağıran taraf (PaymentService) somut sınıfı bilmez, yalnızca arayüze konuşur → ödeme algoritmaları birbirinden ve service'ten izole.

Registry nasıl otomatik topladı?
@Component
public class PaymentStrategyRegistry {
    private final Map<PaymentMethod, PaymentStrategy> strategies = new EnumMap<>(PaymentMethod.class);

    public PaymentStrategyRegistry(List<PaymentStrategy> strategyBeans) {   // ← (1)
        for (PaymentStrategy s : strategyBeans) {
            if (strategies.put(s.getPaymentMethod(), s) != null) {          // ← (2)
                throw new IllegalStateException("Duplicate strategy for " + s.getPaymentMethod());
            }
        }
    }
    public PaymentStrategy resolve(PaymentMethod method) {                  // ← (3)
        PaymentStrategy s = strategies.get(method);
        if (s == null) throw new IllegalArgumentException("No strategy for " + method);
        return s;
    }
}
Spring, List<PaymentStrategy>'e o tipteki TÜM @Component bean'leri enjekte eder. ApplePayPaymentStrategy'yi yazıp @Component koyduğum an, bu listeye otomatik girdi — registry'ye elle kayıt yok.
Her stratejiyi kendi getPaymentMethod()'una göre EnumMap'e indeksler. (Aynı yönteme iki strateji gelirse fail-fast IllegalStateException — sessiz override yok.)
resolve(method) çalışma zamanında O(1) doğru stratejiyi verir.
Bir APPLE_PAY isteği geldiğinde (runtime)
POST /api/orders {paymentMethod:"APPLE_PAY", ...}
  → OrderService.createOrder (değişmedi)
     → PaymentService.charge(APPLE_PAY, total)
        → registry.resolve(APPLE_PAY)         → ApplePayPaymentStrategy  (EnumMap'ten)
        → strategy.pay(total)                  → PaymentResult.success("AP-…")
     → Payment(COMPLETED, ref) → order.attachPayment(...)
Yönlendirme kodu hiç değişmeden yeni yöntemi işledi.

Güvenlik ağı: completeness testi neden otomatik büyüdü?
PaymentRegistryCompletenessTest @EnumSource(PaymentMethod.class) kullanıyor — yani enum'u okur. APPLE_PAY ekleyince test otomatik 3'ten 4 vakaya çıktı ve resolve(APPLE_PAY)'in null olmadığını doğruladı. Tersi senaryo da kıymetli: enum'a değer ekleyip stratejiyi yazmayı unutsaydım, bu test anında patlardı (resolve IllegalArgumentException atar). Yani OCP'yi bir testle zorluyoruz — "yeni yöntem mevcut kodu kırmaz" sözünü makineyle garanti altına aldık.

İskelet + üç odak + canlı OCP örneği oturdu. İstersen Adım 4 — SOLID & design-pattern gözden geçirmesi (her şey doğru yerde mi, gereksiz/eksik soyutlama var mı, SRP/DIP/ISP nerede görünüyor) ile devam edelim; yoksa burada Strategy/Registry üzerine "mülakatçı sorusu" pratiği yapabiliriz. Nasıl istersin?

Adım 3 de oturdu, Open/Closed'ı canlı görmek iyiydi. Şimdi Adım 4. SOLID ve design pattern açısından bir gözden geçirme yapalım. Projede beş SOLID prensibinin her birinin nerede görüldüğünü somut dosyalarla göster. Mülakatta "burayı neden böyle kurguladın" diye soracakları için kararları savunabilmek istiyorum; o yüzden güçlü yanlar kadar zayıf/tartışmalı noktaları da açıkça söyle, "şöyle de yapılabilirdi ama şu yüzden böyle seçtim" diyebileyim. Adım 4 — SOLID & Design Pattern gözden geçirmesi
Beş prensip, somut dosyalarla
S — Single Responsibility
Her sınıfın tek değişme sebebi var: OrderController (HTTP), OrderService (sipariş use-case + tx sınırı), ProductService (ürün ops), PaymentService (ödeme dispatch→entity), NotificationService (bildirim), BulkOrderService (paralel orkestrasyon), OrderMapper/ProductMapper (çeviri), GlobalExceptionHandler (hata→HTTP). En net örnek: paralel dağıtımı OrderService'e tıkıştırmayıp BulkOrderService olarak ayırmak.

O — Open/Closed
PaymentStrategy + PaymentStrategyRegistry (Adım 3'te canlı gördük). Yeni yöntem = yeni @Component, dispatch kodu değişmez.

L — Liskov Substitution
PaymentService, hangi strateji olduğunu bilmeden PaymentStrategy arayüzüyle çalışır — instanceof/downcast yok. CreditCardPaymentStrategy limit kontrolü ekler ama sözleşmeyi bozmaz: normal redde exception fırlatmaz, PaymentResult.failure döner (arayüzün sözleşmesi bu). Yani her strateji yerine geçebilir.

I — Interface Segregation
Bizim yazdığımız arayüz PaymentStrategy minik ve odaklı (2 metot) — kimse kullanmadığı metoda bağımlı değil. Repository'lerde de sadece ihtiyacımız olan sorguları açıyoruz (findByIdForUpdate, findWithDetailsById).

D — Dependency Inversion
Anlamlı sınırlarda soyutlamaya bağlanıyoruz: PaymentService → PaymentStrategy (somut stratejilere değil); registry List<PaymentStrategy> enjekte eder; repository'ler Spring Data arayüzleri. Yüksek seviye (OrderService) düşük seviye detaya (somut ödeme/persistence) değil, soyutlamalara bağlı.

Dürüst değerlendirme
Güçlü ve gerçekten yerinde
Strategy + Registry: Case'in açıkça istediği genişletilebilirliğin birebir karşılığı; üstelik Strategy+Registry'de durduk — AbstractFactory, SPI, ayrı config eklemedik. Gösteriş yok.
PaymentResult value object: "Normal red"i (CC limiti) beklenmedik hatadan ayırır; stratejiyi exception'sız tutup kararı PaymentService'e bırakır. Temiz LSP sözleşmesi.
Rich domain: invariant'lar entity'de (decreaseStock, OrderItem.of). Servisler ince orkestratör.
@EntityGraph, bounded executor, externalized properties: hepsi somut bir sorunu çözüyor (lazy hatası, commonPool starvation, configurability), "havalı olsun" diye değil.
Zayıf / tartışmalı (savunmanı buradan kur)
1. (En önemlisi) @Setter invariant'ı sızdırıyor.
Product.decreaseStock "stok negatif olamaz"ı korur — ama Lombok @Setter, setStockQuantity(int) de üretir; teoride biri bununla stoğu doğrudan (hatta negatife) set edip guard'ı bypass edebilir. Rich-domain iddiasını zayıflatan tek gerçek nokta bu.

"Şöyle de yapılabilirdi": Toptan @Setter yerine yalnızca gereken mutator'ları (ör. bidirectional için OrderItem.setOrder) bırakıp gerisini kaldırırdım; o zaman invariant tamamen kapanırdı. Kısalık/JPA kolaylığı için @Setter bıraktım, ama farkındayım — istersen şimdi sıkılaştırabiliriz.

2. BulkOrderResult (service) ↔ BulkOrderItemResponse (api) neredeyse aynı.
İki benzer kayıt var.

"Şöyle de yapılabilirdi": Controller doğrudan BulkOrderResult döndürebilirdi — ama o zaman API katmanı service tipine bağlanırdı. Ayrı response + toBulkResponse (total/succeeded/failed özetini de ekliyor) bu bağımlılığı kesiyor. Bilinçli katman; ama "ince çoğaltma" diye eleştirilebileceğini kabul ediyorum.

3. Service'lerin arayüzü yok (DIP nüansı).
OrderController somut OrderService'e bağlı, arayüze değil.

"Şöyle de yapılabilirdi": OrderService arayüzü + OrderServiceImpl. Ama tek implementasyon için arayüz açmak "her şeye interface" anti-pattern'i / spekülatif genellik olurdu. DIP'i soyutlamanın kazanç sağladığı yerlerde uyguladım (ödeme stratejileri, repository'ler), gereksiz yere değil. Bu bilinçli bir karar, eksiklik değil — ve mülakatta savunması güçlü.

4. ISP'de çerçeve maliyeti.
Repository arayüzlerimiz ince ama JpaRepository'yi extend ettikleri için aslında onlarca metot (save/delete/findAll…) miras alıyorlar — hepsini kullanmıyoruz.

Bu bizim tasarım kusurumuz değil, Spring Data'nın idiyomatik bedeli. İstersek Repository/CrudRepository'den daha dar bir taban seçebilirdik; pratikte kazanç düşük olduğu için JpaRepository'de kaldım.

5. CreateOrderCommand inceliği.
"Request DTO yetmez miydi?" denebilir. Yetmezdi: service'i web tipinden bağımsız tutuyor (service api paketini import etmemeli). İnce ama gerçek bir sınır.

Net verdict (eklenecek / çıkarılacak)
Çıkarılacak pattern: yok. Hiçbir pattern gösteriş için değil; restraint (Template Method/Observer/Saga/Factory eklememek) doğru.
Eksik kalan prensip: yok. DIP'in service'lere uygulanmaması bilinçli ve doğru; ISP'deki gevşeklik çerçeve kaynaklı.

README'deki "Transaction / rollback senaryosunu test etme" bölümüne PowerShell manuel kontrolünü eklemişsin ama şu an sadece stok kaynaklı rollback örneği var (Mouse+Pen → 409, MultiItemOrderTest'in karşılığı). Bu bölümde iki otomatik test listeleniyor; ikincisinin (OrderServiceRollbackTest — ödeme patlayınca rollback) de canlı bir karşılığı olsun istiyorum.
Mevcut 409 örneğinin hemen altına, ödeme kaynaklı rollback için ikinci bir PowerShell bloğu ekle. Bunu canlı test ettim, çalışıyor: limiti aşan bir kredi kartı siparişi → 422 → stok geri sarılır:

# Ödeme kaynaklı rollback (OrderServiceRollbackTest'in canlı karşılığı):
# Laptop(id=1) x7 = 10.499,93 > 10.000 kredi kartı limiti → ödeme reddi (422)
'{"customerId":7,"paymentMethod":"CREDIT_CARD","items":[{"productId":1,"quantity":7}]}' | Out-File -Encoding ascii body.json
curl.exe -s -i -X POST http://localhost:8080/api/orders -H "Content-Type: application/json" --data "@body.json"
# rollback kanıtı: Laptop stoğu hâlâ 25 olmalı
(Invoke-RestMethod http://localhost:8080/api/products/1).stockQuantity
İki örneği kısaca etiketle ki hangisi hangi testin karşılığı belli olsun: ilki stok kaynaklı (409) → MultiItemOrderTest, ikincisi ödeme kaynaklı (422) → OrderServiceRollbackTest. Küçük bir dürüst not da ekleyebilirsin: OrderServiceRollbackTest testin içinde ödemeyi @MockBean ile patlatıyor; manuel versiyonda ise aynı ödeme-patlaması + rollback gerçek kredi kartı limiti aşılarak tetikleniyor (davranış aynı, tetikleme farklı). Otomatik test komutu (mvn test -DforkCount=0 "-Dtest=OrderServiceRollbackTest,MultiItemOrderTest") zaten var, o kalsın.
